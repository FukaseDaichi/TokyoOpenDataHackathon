# ③ 使えるオープンデータの棚卸し

> AIでアイディアを練るときの「データの裏付け」のインプット。都カタログ＋外部データの使えるものをここに集約。
> 2026-07-10 再調査（サブエージェント3本・CKAN API実測）を反映済み。

- 最終更新: 2026-07-10
- 主な入口: [ポータル](https://portal.data.metro.tokyo.lg.jp/)／[カタログ(CKAN)](https://catalog.data.metro.tokyo.lg.jp/)／[API仕様検索](https://spec.api.metro.tokyo.lg.jp/spec/search)／[活用事例](https://portal.data.metro.tokyo.lg.jp/case/)

## カタログ全体像

- 約54,000データセット、うち約35,000CSVがAPI化。原則 **CC BY 4.0**。基盤はCKAN、区市町村データも横断集約。
- カテゴリ別件数（多い順）: **行財政 3,891**／くらし・住まい 1,142／子供・若者・教育 827／インフラ・まちづくり 806／産業・仕事 709／高齢者・福祉 703／環境・自然 615／健康・医療 524／文化・芸術 471／防災 432／…／交通 79（動的交通はODPT側が本体）。
- **CKAN API（`/api/3/action/package_search` 等）は直接利用可**（実測確認済み。WebはBot遮断403だがAPIは生きている）→ カタログ横断検索そのものを部品化できる。
- カタログ側の改善: 全文検索機能（2026-02導入）＋掲載データ一覧CSV（2026-03更新）。
- **最大の弱点: 多くのCSVに緯度経度がない** → ジオコーディング/ポリゴン結合の前工程を時間予算に計上（後述のABRで低コスト化可能）。

## データ選定の品質ゲート（全候補に適用）

1. **既存ダッシュボード（公式可視化含む）で答えられない「新しい問い」があるか**。なければ捨てる。
2. **2時間 kill test**: 20行読む／地図に出す／欠損率／更新日／結合キーを確認。失敗したら即捨て。
3. **倫理設計が成立するか**: 個人を推定しない集計粒度・地域スティグマ回避・「推奨」でなく「判断支援」。
