# 区詳細ページ リデザイン設計書

作成日: 2026-07-12
対象: `/ward/[slug]/`（23区分の静的ページ）

## 1. 目的

区詳細ページを「診断結果の補足」から「区を知るための図鑑本編」へ拡張する。

1. **位置理解**: 区が東京都のどこにあるかを3Dマップで直感的に見せる。
2. **社会的意義**: パラメータだけでなく、区の政策（基本構想・基本計画の柱）と裏付け統計を載せ、オープンデータで区の実像と志を伝える。
3. **トーン**: 絵本図鑑（羊皮紙＋金インク）の世界観を維持する。

## 2. 決定事項

- 3Dマップは**演出＋位置理解**に限定する。インタラクティブな区間移動やデータ可視化（コロプレス）はしない。
- 政策情報は**手動キュレーション**。各区の基本構想・基本計画から「政策の柱」を3〜5個要約し、出典URL付きで同梱する（横断的な機械可読オープンデータは存在しないことを調査で確認済み）。
- デザイントーンは現行の絵本図鑑トーンを維持する。
- `output: 'export'` の静的サイト制約は変えない。データはすべてビルド時同梱。

## 3. ページ構成

上から順に。①③⑦は現行維持、②⑤⑥が新規、④が拡充。

| # | セクション | 内容 |
|---|---|---|
| ① | ヘッダー | 肖像・系統・名前・キャッチ（現行維持） |
| ② | 東京のどこにいる？ | 3D東京都マップ。詳細は§4 |

| ⑥ | 区のプロフィール | 区章SVG・区の花/木/鳥・人口・面積・区制施行年などの図鑑的基本情報カード |
| ⑦ | 主要駅・なかま・出典 | 現行維持。出典リストに新データ源を追記 |

## 4. 3D東京都マップ

### 4.1 表現

- 羊皮紙に広げた「魔法の地図」。23区を金インク（`#b8923f`）の細線ポリゴンで描く。
- 当該区のみテーマカラー（`wardTheme(code).color`）で塗り＋発光＋押し出しで浮上。
- 区役所位置に封蝋風のピン。隣接区の区名を薄墨（`#7a5c2e`）で小さく表示。
- カメラは俯瞰固定でゆっくり自動回転（±数度の揺らぎ）＋マウス微追従。スクロールは奪わない。
- 飾り: 方位記号・経緯線風の飾り罫を金インクで。

### 4.2 実装

- `src/ui/WardMap3D.tsx`（新規）: R3F。`next/dynamic` + `ssr: false` で読み込み、`ExtrudeGeometry` で各区を薄い板状に押し出す。当該区のみ高さと発光を強調。
- 品質判定はヒーローの `src/hero/quality.ts`（`detectQuality()` / `QualityTier`）を流用。`fallback` 時と Canvas 初期化失敗時（エラーバウンダリ）は 2D SVG 地図 `WardMap2D.tsx` へ。
- `WardMap2D.tsx`（新規）: 同じ TopoJSON から生成した SVG パスで平面地図を描画。当該区をテーマカラーで塗る。`prefers-reduced-motion` 環境の既定表示でもある。
- ジオメトリ生成（経緯度→ローカル座標変換、`THREE.Shape` 構築）は副作用のない純ロジック `src/lib/geo.ts` に置き、Vitest を先に書く。

### 4.3 ジオデータパイプライン

- 出典: 国土数値情報 行政区域データ N03（2025年版、政府標準利用規約・商用可）。`data/raw/` に取得物を置く。
- 簡略化はトポロジー整合性のため smartnews-smri/japan-topography の簡略化済み TopoJSON（N03 由来、隣接区の境界線が共有アークで一致）を入力に使う。SmartNews 側はクレジット不要、国土数値情報の出典表記のみ必要。
- `data/build_geo.py`（新規、標準ライブラリのみ）: TopoJSON をデコードして23区を抽出し、東京中心の局所平面座標（km）へ投影 → `data/processed/ward-geo.json`（目標 80KB 以下）を生成。`src/data/ward-geo.json` へコピーして同梱（wards.json と同じ二重管理方式）。
- 区の代表点（封蝋ピン位置）は最大リングの重心をビルド時に計算して同ファイルに収録する（外部の区役所位置データは使わない）。区面積 `area_km2` も同時に計算し、プロフィールカードで使う。
- 不変条件: 23区分そろえる。区コード順 13101→13123。`data/processed/` は手編集しない。

## 5. データ拡充

### 5.1 手動キュレーション（新規 `src/data/ward-policies.json`）

手動管理ファイル（Pythonジェネレーター対象外）。形式:

```json
{
  "13103": {
    "emblemNote": "区章の由来ひとこと（任意）",
    "flower": "バラ", "tree": "ハナミズキ", "bird": "ユリカモメ",
    "policies": [
      { "title": "国際性を育むまちづくり", "summary": "2文程度の中立的な要約。", "source": "港区基本計画", "url": "https://..." }
    ]
  }
}
```

- 政策の柱は各区公式の基本構想・基本計画から要約。**中立・前向きな表現**とし、区間の優劣を示唆する記述はしない。
- 収集はClaudeが23区分を起草し、ユーザーがレビューして確定する。
- 区章SVGは Wikimedia Commons のパブリックドメイン素材を `public/emblems/<slug>.svg` に配置。区章条例に配慮し中立的表示に限定。出典はページの出典欄に明記。

### 5.2 新統計（`data/build_details.py` に追加）

財政力指数・1人当たり公園面積・外国人人口比率は既に `WardMetrics` / `ward-details.json` に実装済みのため対象外。新規は次の3種（すべて23区で揃うことを調査済み）。

| 統計 | 出典 | 形式 |
|---|---|---|
| 平均所得（納税義務者1人当たり課税対象所得） | 総務省 市町村税課税状況等の調（e-Stat） | CSV |
| 犯罪認知件数（人口千人当たりに正規化） | 警視庁（都カタログ CC BY 4.0） | CSV・町丁別→区集計 |
| 待機児童数 | 東京都 保育サービス状況 | Excel |

あわせて、プロフィールカード用に総人口（`build_details.py` 内で算出済みの `_total_population()` を出力に追加）と区面積（§4.3 でジオデータから算出）を公開する。

- 各統計は `ward-details.json` に追記し、`buildWardStats` 経由で StatBar 表示。犯罪件数など負のラベルになり得る指標は**中立的な文言**（例:「安心づくりのデータ」）で提示し、順位の見せ方に注意する。
- 5軸ベクトルの算出には**組み込まない**（診断ロジックは不変）。

## 7. エラーハンドリング・フォールバック

- WebGL不可・reduced motion・Canvas初期化失敗 → 2D SVG 地図（既存ヒーローと同じ三段構え）。
- `ward-policies.json` に当該区の政策が未収録でもページは壊れない（セクションを出さない）。
- ジオデータ欠損はビルド時に検出（23区分の存在チェックをテスト化）。

- `src/data/ward-geo.json` / `ward-policies.json`: 23区分そろい・区コード順・URL形式のデータ健全性テスト。
- `WardPage`: 新セクションのレンダリングテスト（政策なし区の非表示分岐含む）。

## 9. スコープ外（第2フェーズ候補）

- 名所・風景写真（Wikimedia Commons のライセンス個別確認が必要なため今回は見送り）
- 3Dマップのインタラクション（隣接区クリック遷移など）
- 避難所・子育て施設など地物の地図上プロット

## 10. リスク

- N03 ポリゴンの簡略化しすぎで区形が崩れる → 簡略化率を段階的に確認（目標: TopoJSON 50KB 以下かつ見た目維持）。
- 政策要約の中立性 → レビュー必須。地域スティグマにつながる表現を避ける既存方針に従う。
- 犯罪統計の見せ方 → 順位を強調しない・文言を中立に。懸念が残る場合は掲載自体を再検討し `07-risks-and-concerns.md` に記録。
- モバイル性能 → 品質ティア low で DPR・ジオメトリ解像度を抑制。ヒーローと同居しないページなので負荷は限定的。
