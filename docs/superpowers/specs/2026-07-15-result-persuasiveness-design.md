# 診断結果の納得感強化 設計書（2026-07-15）

## 背景と課題

診断結果ページ（`src/ui/pages/ResultPage.tsx`）は、診断で算出したユーザーの5軸ベクトルを sessionStorage に保存しているが、画面には一切表示していない。現状ベクトルは相性ランキングの並び替えと「にてる度%」の算出にしか使われていない。

このため次の2点が欠けており、「にてる度81%」と言われても何がどう似ているのか伝わらない。

1. 自分がどんなタイプだったのかが分からない
2. なぜこの区と相性がいいのかが分からない

さらに「なるほど！」の後に区そのものへ興味を持たせる導線が、既存の相性ランキングと区詳細リンクしかない。

## ゴール

診断者が結果ページで次の順に体験できること。

1. **自分を知る** — 自分のタイプ名と5軸の形が見える
2. **納得する** — 区ちゃんとの一致点が視覚とデータ根拠つきの文章で示される
3. **興味を持つ** — 一致点に関連する区の実データと政策に触れ、区詳細・公式サイトへ進みたくなる

## スコープ

- 結果ページへの新セクション追加（下記①②③）と既存 `Radar` の2ポリゴン対応
- タイプ命名・一致軸選定の純ロジック追加（`src/lib/`、Vitest 先行）
- AI事前執筆の相性文データ（5軸×23区=115本）の同梱
- `ward-policies.json` の政策エントリへの軸タグ付けキュレーション
- 関連するシステム設計書の更新

### 非スコープ

- 結果ページへの地図（WardMap）組み込み
- 名前付き固定タイプ（16タイプ診断方式）
- Xシェア文言の変更
- 診断ロジック（質問・採点・校正割り当て）の変更

## ① あなたのタイプ

### タイプ命名ロジック（`src/lib/personaType.ts`・新規純ロジック）

- 入力：ユーザーの `AxisVector`（各軸 [-1, 1]）
- 絶対値が閾値 **0.3 以上**の軸を対象に、絶対値降順で上位2軸を選ぶ。同値は `AXIS_KEYS` 順で先勝ち
- 各軸の極（5軸×2方向=10極）に呼び名と説明文を定義し、「{極A}×{極B}タイプ」と命名する
- 対象軸が1つなら「{極A}タイプ」、0なら **「バランスタイプ」** にフォールバック

極の呼び名（ドラフト。実装時に語感を調整してよいが、中立・前向きの原則を守る）：

| 軸 | 高い側 | 低い側 |
| --- | --- | --- |
| 賑わい | 都会派 | のんびり派 |
| 成熟 | おとな志向 | フレッシュ志向 |
| みどり | みどり派 | シティ派 |
| 世帯 | ファミリー志向 | ソロ充 |
| 華やぎ | 華やか志向 | 堅実派 |

- 各極に1文の説明を持ち、上位2極の説明を組み合わせてタイプ説明文（1〜2文）を生成する。バランスタイプにも専用説明文を用意する

### レーダー重ね表示（`src/ui/Radar.tsx` 拡張）

- `Radar` にオプションの第2ベクトル（オーバーレイ）を追加し、区ちゃんのポリゴン（テーマカラー）にユーザーのポリゴン（破線・ニュートラル色）を重ねる
- 「あなた」「{区名}ちゃん」の凡例を表示する
- 既存の単独利用（区詳細ページ等）の見た目は変えない

## ② なぜ相性がいいの？

### 一致軸選定ロジック（`src/lib/personaType.ts` に同居、または `matching.ts` 拡張）

- ユーザーと区の軸差 `|user[k] - ward[k]|` が小さい順に、ユーザー側の絶対値が 0.3 以上の軸から上位2軸を「一致軸」とする
- 条件を満たす軸が2未満の場合は、絶対値条件を外して差の小さい順に補充する（常に2軸を返す）

### 表示

- 5軸すべてについて、`AXIS_LABELS` の low/high ラベル付きバー上に「あなた」と「区ちゃん」の2マーカーを置いた比較リストを表示し、一致軸2つをハイライトする
- 一致軸1位について、AI事前執筆の相性文（下記）を表示する

### 相性文データ（AI事前執筆・5軸×23区=115本）

- `src/data/ward-affinity.json` ＋ローダー `src/data/affinity.ts`（`rationale.ts` と同方式の同梱データ）
- キー構造：`{ [wardCode]: { [axisKey]: string } }`。全23区×5軸を必須とする
- 各文は2〜3文。**その軸でユーザーと区が近い**前提で書く（文の方向は区の軸値の符号に合わせる。例：新宿の世帯軸は低い→「一人の時間を大切にするあなた」向けの文）
- 区の実データの数値・順位（例：単身世帯率67.8%で23区1位）を盛り込む。数値は執筆時点のスナップショットである旨をファイル冒頭コメントに明記する
- トーンは中立・前向き。低い軸を否定的に書かない（AGENTS.md の原則）
- 表示側に `rationale` と同様のAI執筆バッジを付ける

## ③ 実はこんな街

### データカード（2枚）

一致軸2つのそれぞれに対応する基本指標を、値と「23区N位」バッジ付きのカードで表示する。順位は既存の `buildRadarStats` ＋ `rankOf` を再利用する。

| 一致軸 | 表示指標 |
| --- | --- |
| 賑わい | 昼夜間人口比率 |
| 成熟 | 高齢化率 |
| みどり | 一人当たり公立公園面積 |
| 世帯 | 区の世帯軸が正なら子育て世帯率、負なら単身世帯率 |
| 華やぎ | 財政力指数 |

カード群の下に区詳細ページへの導線（既存の「より詳しく見る」と統合してよい）を置く。

### 政策紹介（1件）

- `ward-policies.json` の各政策エントリにオプションの軸タグ `axes?: AxisKey[]` を追加できるよう `src/data/policies.ts` の型を拡張する
- 既存23区分の政策にキュレーションパスで軸タグを付ける（AI補助で下書き→内容確認）
- 表示選択：一致軸とタグが交差する最初の政策を優先し、なければ先頭の政策。政策が無い区（データ欠損時）はこのブロックを非表示
- 「{区名}ちゃんはこんなことを頑張ってる」の見出しで、政策タイトル・要約・出典名＋公式URLリンクを表示する

## エラー・エッジケース

- sessionStorage が読めない／resultCode がページの区と不一致 → 既存どおり受け手表示（①②③非表示）
- ユーザーベクトルが全軸ほぼ0 → タイプは「バランスタイプ」、一致軸は差の小さい順で2軸選定し通常表示
- 相性文・政策データの欠損 → 該当ブロックのみ非表示にし、ページ全体は壊さない

## テスト方針（Vitest 先行）

- `personaType.test.ts` — 命名（2軸／1軸／バランス）、閾値境界（0.3）、同値タイブレーク、一致軸選定と補充
- `affinity.test.ts` — 全23区×5軸の相性文が存在し空でないこと（`rationale.test.ts` と同型）
- `policies.test.ts` 拡張 — 軸タグが `AXIS_KEYS` の値のみであること
- `ResultPage.test.tsx` 拡張 — 診断者に①②③が出ること、受け手に出ないこと、一致軸に応じたカード・政策の選択
- `Radar` のオーバーレイ描画は既存テスト方式に合わせてスナップショットまたは要素存在で検証
