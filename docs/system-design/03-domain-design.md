# ドメイン・診断設計

## 1. ドメインモデル

5軸のキーと順序は `src/domain/axes.ts` の `AXIS_KEYS` を正とする。

| 軸キー | 負方向 | 正方向 | 表示名 |
|---|---|---|---|
| `liveliness` | 静か | 賑やか | 賑わい |
| `maturity` | 若い | 成熟 | 成熟 |
| `greenery` | 都会的 | みどり | みどり |
| `family` | おひとりさま | ファミリー | 世帯 |
| `luxury` | 堅実 | 華やか | 華やぎ |

`AxisVector` は5軸それぞれの数値を持つ。区ベクトルと診断ベクトルはともに原則 `[-1, 1]` の範囲で扱う。`Ward` は区コード、区名、5軸、系統、根拠指標を持つ。

## 2. 区ベクトルの生成

23区の各指標を、区集合内の最小値・最大値で `[-1, 1]` に変換する。

```text
signedMinMax(x) = 2 × (x - min) / (max - min) - 1
```

最大値と最小値が同じ場合は全区を0とする。

| 軸 | 正規化前の合成値 | 変換 |
|---|---|---|
| 賑わい | 昼夜間人口比率 | `log(x)` 後にsigned min-max |
| 成熟 | 高齢化率 − 年少人口率 | signed min-max |
| みどり | 一人当たり公立公園面積 | signed min-max |
| 世帯 | 子育て世帯率 − 単身世帯率 | signed min-max |
| 華やぎ | 財政力指数 | signed min-max |

賑わいだけ対数を取るのは、昼夜間人口比率の最大値が他区から大きく離れており、線形変換では区間の差が潰れるためである。対数関数は正値を前提とする。

## 3. 質問と採点

質問定義は `src/lib/quiz.ts` の `QUESTIONS` に置く。各選択肢は1軸または2軸へ `+1` / `-1` を与える。

| 軸 | 対象質問数 |
|---|---:|
| 賑わい | 3 |
| 成熟 | 2 |
| みどり | 2 |
| 世帯 | 2 |
| 華やぎ | 3 |

各軸は加算値ではなく、その軸に回答効果があった質問数で割った平均値とする。これにより質問数が異なっても全軸を同じ `[-1, 1]` で比較できる。不正または不足した回答はその質問を無視し、寄与がない軸は0とする。通常UIは10問すべての有効な回答だけを渡す。

## 4. マッチング、配分校正、類似度

診断ベクトル `u` と区ベクトル `w` の距離は、5軸を同じ重みで扱うユークリッド距離である。

```text
distance(u, w) = √Σ(axis)(u_axis - w_axis)²
```

距離の昇順を未校正の順位とする。同距離の場合は元の区配列順、すなわちJIS区コード順が維持される。実際の診断結果は `src/data/diagnosis-assignments.json` の校正済み割り当てを使い、単純な距離1位だけでは決めない。

割り当てスナップショットは `npm run build:diagnosis` で全回答パターンを列挙して生成する。10問・各2択の現行構成では1024通りであり、次の制約を順に満たす。

- 各区の割り当てを `ceil(1024 × 1%) = 11` 件以上とする。
- 各区の割り当てを `floor(1024 × 10%) = 102` 件以下とする。
- 割り当て先を未校正の距離順位5位以内に限定する。
- 下限未達区、次に上限超過区へ、距離の増加が最小となる回答パターンを決定的に移す。

生成後の配分は全23区が11〜102件、最大の未校正順位は5位となる。割合は全回答パターンを同確率と仮定した設計上の比率であり、実利用者の回答分布を保証するものではない。割り当てキーは平均後の5軸ベクトルではなく回答番号列であるため、同じ5軸ベクトルへ集約される異なる回答列が別の区になる場合がある。

実行時は回答番号列をmixed-radixの整数へ変換し、スナップショットの配列を引く。質問ID、選択肢数、区順序の変更時はスナップショットを再生成し、同期テストで差分を検出する。結果ページのランキングは校正結果を先頭へ置き、残りを未校正の距離順で表示する。

表示用の「にてる度」は、5軸すべてが最大差2となる理論最大距離 `√20` を0%として線形変換し、整数へ丸める。

```text
similarity = round(max(0, 1 - distance / √20) × 100)
```

この値は説明用の距離尺度であり、統計的確率ではない。

相性ランキングの表示%は `compatibilityPercents`（`src/lib/matching.ts`）で導出する。診断割り当ては校正済みで結果の区が距離最小とは限らないため、生の%の最大値が結果の区の%以上になる場合だけ、最大が「結果% − 1」となるよう全体を比例スケーリングし、表示が常に「結果 > 相性1位 ≥ 2位 ≥ 3位」の単調な並びになることを保証する。逆転がない場合は生の値のまま表示し、校正済み割り当てとX共有文面の%には影響しない。

### タイプ命名と一致軸選定

`src/lib/personaType.ts` は結果ページの「あなたのタイプ」と「相性の理由」を支える純ロジックである。

- `personaType(u)`: 絶対値が閾値0.3以上の軸を絶対値降順（同値は `AXIS_KEYS` 順）に並べ、上位2軸の極（各軸の高低2方向に定義した呼び名。例: 賑わい高=都会派、世帯低=ソロ充）を「{極A}×{極B}タイプ」と命名する。該当軸が1つなら単軸名、0なら「バランスタイプ」を返す。説明文も極ごとの一言から合成する。
- `selectMatchedAxes(u, w)`: 軸差 `|u_axis - w_axis|` の昇順で、利用者側の絶対値が0.3以上の軸から一致軸2つを選ぶ。2軸に満たない場合は絶対値条件を外して差の昇順で補充し、常に2軸を返す。

呼び名と説明文は区の表現と同じく中立・前向きの原則に従い、低い側の極にも否定的な語を使わない。表示層では選定軸でも軸差が0.5を超えるものに「一致」バッジを付けない。

## 5. 系統クラスタリング

`src/lib/cluster.ts` の決定的k-meansで23区を6系統へ分ける。

- クラスタ数: 6
- 初期重心: JIS区コード順の先頭6区
- 距離: 診断と同じ5軸ユークリッド距離
- 更新回数: 20回固定
- 乱数: 使用しない
- 空クラスタ: 直前の重心を維持
- 表示名: `系統1` から `系統6`

入力スナップショットが同じなら結果は同じになる。一方、番号自体に意味はなく、初期重心またはデータが変われば所属と番号が変わり得る。

## 6. 順位と平均比

区詳細の順位は、対象値より大きい区の数に1を加えて算出する。既定は値が大きいほど上位である。待機児童数はUI層で符号を反転してから順位と平均比を求めるため、少ないほど上位になる。平均比は対象値を23区平均で割り、平均が0の場合だけ0を返す。

## 7. ドメイン不変条件

- `AXIS_KEYS` は5件で、順序を描画・距離・テストで共有する。
- 基本スナップショットは23区すべてについて7つの根拠指標を持つ。
- 区コード、slug、キャラクターマスターは23区を一対一で対応させる。
- 診断結果の全回答パターン配分は各区1%以上10%以下で、未校正の距離順位5位以内に収まる。
- `diagnosis-assignments.json` は質問定義と区データからの決定的な再生成結果と一致する。
- 正規化後の区ベクトルは全軸 `[-1, 1]` に収まる。
- 同じスナップショットから生成する系統は決定的である。

## 8. 区詳細ページの周辺データ型

診断・マッチングの中核である `Ward` / `AxisVector` に加え、区詳細ページ（`/ward/[slug]/`）は次のデータ層の型を参照する。いずれも5軸ベクトルの正規化には関与しない付帯情報であり、型定義とローダーの詳細は [04-data-design.md](04-data-design.md) を正とする。

| 型 | 定義箇所 | 内容 |
|---|---|---|
| `WardGeo` | `src/data/geo.ts` | 区境界（外周リングのみ、面積降順）、重心、面積。地図描画専用で診断ロジックからは独立 |
| `WardDetails` | `src/data/details.ts` | 地価、人口、平均所得、外国人人口比率、主要駅、人口千人当たり刑法犯認知件数、待機児童数。型上は人口と地価以外が任意で、現在のスナップショットは主要駅を除く全項目を23区分持つ |
| `WardProfile` | `src/data/policies.ts` | 区の花・木・鳥（各項目は複数指定に対応する配列）と政策キュレーション（`policies[]`、最大5件）。`src/data/ward-policies.json` を直接手編集する非生成データ |

現在の `ward-policies.json` は23区すべてにプロフィールと政策を収録している。ただし型とローダーは部分収録を許容し、未知または未収録の区では `loadWardProfile()` が `null` を返す。花・木・鳥の各項目も任意で、未制定・未収録の項目だけをUIから省く。区章は `WardProfile` ではなく、slugに対応する `public/emblems/{slug}.svg` を区詳細ページが直接参照する。
