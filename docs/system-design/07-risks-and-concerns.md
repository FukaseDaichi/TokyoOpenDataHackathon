# 現行コードの懸念事項

この文書は、現行実装を正としてコード、設定、データ、テスト、生成物を照合した時点のリスクと保守上の注意をまとめる。将来機能の一覧ではなく、現在の正しさ、再現性、運用性に影響する事項だけを対象とする。

## 優先度の定義

- **P1**: 公開結果の誤り、配信不備、データ不整合、権利確認不能につながるため先に解消したい
- **P2**: 現在は動くが、更新・保守・利用環境によって問題化しやすい
- **P3**: 品質改善として有効だが、直ちに機能を壊さない

## 懸念一覧

| 優先度 | 懸念 | コード上の根拠 | 影響 | 推奨対応 |

| P1 | 配信用サイトURLの欠落がビルド失敗にならない | `next.config.ts` は本番時に警告するだけで、`app/layout.tsx` は `NEXT_PUBLIC_SITE_URL` がない場合に `metadataBase` を設定しない | 本番ビルドが成功してもOGPの絶対URLがlocalhost基準になり、Xカードが表示されない | CI/デプロイ用ビルドでは環境変数を必須化し、公開HTMLの `og:image` がHTTPSの公開ホストを指すことを検査する |
| P1 | 集計JSONと配信用JSONが手動コピー | `data/processed/*.json` と `src/data/*.json` の間は `cp` で同期し、生成・ビルド時の自動差分検査がない | データを再集計しても、コピー漏れによりアプリが古い値を配信できる | ジェネレーターから配信用JSONも同時生成するか、同期・差分検査をnpm scriptとCIへ組み込む |
| P1 | 区章SVGの取得・権利情報をファイル単位で追跡できない | `public/emblems/*.svg` は23区分あるが、取得URL、取得日、作者、ライセンス、自治体章としての利用条件を対応づける台帳がない。`docs/research/ward-symbols-audit-2026.md` の対象は花・木・鳥と政策であり、SVG自体の来歴ではない | 再配布・提出時に各ファイルの利用根拠を監査できない | slugごとの来歴と利用条件を台帳化し、根拠を確認できないSVGは公開対象から外す |
| P2 | 区マスターが複数箇所に重複 | 区名一覧が `data/build_wards.py`、区コード列が `data/build_details.py` / `data/build_geo.py`、slug・画像・表示属性が `src/hero/wards.ts` に分散する | 区名、コード、slug、画像、ルートの修正時に更新漏れが起きる | 1つの機械可読マスターからTypeScriptとPythonの双方を参照する |
| P2 | Python実行環境が固定されていない | `build_wards.py` と `build_details.py` は `openpyxl` を使うが、requirementsやPythonバージョン定義がない | 別環境でデータを再生成できない、または依存差で結果が変わる | `requirements.txt` 等でPythonと依存バージョンを固定し、クリーン環境での再生成確認を追加する |
| P2 | k-meansの分類が先頭6区と固定20反復に依存する | `src/lib/cluster.ts` は初期重心を入力先頭6区から取り、収束判定をせず20回更新する。表示名は数値ラベルをそのまま `系統1`〜`系統6` にする | データや区順序の更新で所属・番号が大きく変わっても、通常のテストでは意味上の変化を検知しにくい | 現行所属をレビュー可能なスナップショットにするか、初期化・収束条件・命名規則を明示的にする |
| P2 | モーダルのフォーカス管理が不完全 | `WardModal` は閉じるボタンへの `autoFocus` とEscape操作を持つが、フォーカストラップ、背景のinert化、呼出元への復帰、背景スクロール抑止を実装していない | キーボードや支援技術の利用者が背面へ移動し、閉じた後の操作位置を失う | dialogユーティリティ等でフォーカストラップ、復帰、背景抑止をまとめて実装する |
| P2 | 3Dヒーローの区カードをキーボードで選択できない | `HeroCanvas` のカード選択はThree.jsのpointer/clickイベントだけで、DOM上の区名ラベルは操作要素ではない | high/lowティアではヒーロー内の区選択をキーボードで行えない。後段の図鑑では選択できるが、同じ体験にはならない | DOMラベルをボタン/リンク化するか、Canvasと同期するキーボード用操作層を置く |
| P2 | ブラウザベースの配信品質ゲートがない | `.github/workflows/deploy.yml` はtest/build後に配信するが、実ブラウザE2E、リンク検査、アクセシビリティ自動監査、WebGLの実描画検証は行わない | 静的ルートと型・単体テストの回帰は検出できる一方、レスポンシブ、3D/2Dフォールバック、公開後リンクの不具合を見逃しうる | リンク・OGP検査と代表ブラウザE2EをCIへ追加する |
| P2 | アセットの23区完備を一括検証していない | `manifest.test.ts` が確認する配信用画像は512px WebPだけで、896px WebP、キャラクター原本、OGP原本、配信用OGP、トップOGPは検査しない。タイトルとモーダル画像の生成もpackage scriptsへ統合されていない | 一部画像の再生成・配置漏れや原本との不一致が起きる | `build:assets` と検証スクリプトを用意し、slug集合に対する原本・全生成物の存在、寸法、上限サイズを検査する |

- `NEXT_PUBLIC_SITE_URL=https://example.pages.dev npm run build`: 静的エクスポート成功。

- `data/processed/wards.json` と `src/data/ward-metrics.json`、`data/processed/ward-details.json` と `src/data/ward-details.json`、`data/processed/ward-geo.json` と `src/data/ward-geo.json` はバイト単位で一致する。

## 受容している制約

次は欠陥ではなく、静的・無料・個人情報を保持しない構成として意図的に受容している。

- 共有URLから回答内容や相性ランキングを復元しない。
- データ更新には再集計と再デプロイが必要。
- 外部API停止の影響を受けない代わりに、表示値は取得時点のスナップショットである。
- WebGL不可時は演出品質を下げ、診断と図鑑の機能を優先する。
- three.js本体は `HeroCanvas`（トップ）と `WardMap3D`（区詳細）の動的チャンクへ各約384KB複製される。2026-07-19にwebpack splitChunksで共有チャンク `three-vendor` への統合を試したが、チャンクが両者のunion（832KB）になり単一ページ訪問のLighthouse計測が悪化した（トップ94→89、区詳細73→63・転送+359KB）ためリバートした。ページ単位でtree-shakeされた複製の方が初回訪問体験で優位という計測結果に基づき、複製を受容する。
